
# Pre-Launch Bug & Inconsistency Audit

## CRITICAL Issues (Will Break Customer Experience)

### 1. "Daily" vs "Monthly" Terminology Mismatch
The app has a fundamental messaging conflict:
- **Subscription page** (line 351): Shows `"{daily_stories} stories per day"` -- but the actual limit is **monthly** (10/month)
- **Mission.tsx** (line 1008): Says `"Daily Adventure Limit Reached"` -- but the limit is **monthly**, not daily
- **Database plans**: The `daily_stories` field in `subscription_plans` is set to `3` for free and `10` for premium, but these are treated as **monthly** limits in the code
- Customers will be confused whether they get 10 stories per day or per month

### 2. Duplicate/Conflicting Free Plans in Database
There are **two** free-tier plans:
- `"basic"` (story_limit: 3, daily_stories: 3)
- `"free"` (story_limit: 1, daily_stories: 1)

The code defaults to 3 stories for free users, but if a user somehow gets the `"free"` plan assigned, they'd only get 1 story. The `SubscriptionModal` references both. This is confusing and could cause unpredictable behavior.

### 3. Mission Page Not Protected by Auth
`/mission` is **not** wrapped in `ProtectedRoute` in App.tsx (line 92). An unauthenticated user could navigate to `/mission` directly. The Mission page does use `useAuth()` but only optionally -- it will attempt to load stories from the database without a user, hitting RLS errors silently. Story completions and reading tracking will fail.

### 4. Old Brand Name "StoryMaster Quest" Still Present
Despite rebranding to "StoryMaster Kids", the old name appears in:
- **StoryGallery.tsx** (line 157): Shared story text says `"This adventure was generated by StoryMaster Quest!"`
- **support@storymasterquest.com** email in 5 files (PrivacyPolicy, TermsOfService, Support, SubscriptionSuccess) -- this is fine if that's the real email domain, but inconsistent with the "Kids" rebrand

### 5. "Unlimited" Claim on Success Page
**SubscriptionSuccess.tsx** (line 99): Says `"Access unlimited interactive stories every month"` but the actual limit is **10 stories/month** for Adventure Pass. This is misleading and could cause refund disputes.

## HIGH Priority Issues

### 6. Read-to-Me Button Disabled Logic Inconsistency
In Mission.tsx (line 1211), the button is disabled when `!userPlan?.features?.read_to_me`, but the `handleReadToMe` function (line 158) has its own `hasReadToMeAccess()` check that also checks plan name. The tooltip at line 1237 says `"Only Available for Premium"` but it should say **"Adventure Pass Plus"** to match branding.

### 7. Subscription Page Shows "X stories per day" Instead of "per month"
Line 351 of Subscription.tsx displays `daily_stories` value as "stories per day" when the system actually enforces a **monthly** rolling 30-day limit. Premium users would expect 10 stories/day but only get 10/month.

### 8. `cancelSubscription()` Only Cancels by Device ID
The `cancelSubscription()` function in subscription.ts only filters by `device_id`. If a user logs in on a new device, cancellation won't find their subscription. The `getUserSubscription()` function correctly falls back to `user_id`, but cancel doesn't.

### 9. "Continue Quest" Button When No Stories Exist
On Dashboard.tsx, the "Continue Quest" button navigates to `/mission` even when `inProgressStories.length === 0` (line 207-208). This sends the user to Mission without a `?resume` or `?new` param, which could cause unexpected behavior (trying to load a nonexistent story).

## MEDIUM Priority Issues

### 10. Security: 4 Supabase Linter Warnings
- **Overly permissive RLS policy** (`USING (true)`) on at least one table
- **OTP expiry too long** -- should be reduced for production
- **Leaked password protection disabled** -- should be enabled before launch
- **Postgres version needs security patches**

### 11. `verify_jwt = false` on All Edge Functions
All edge functions in `supabase/config.toml` have `verify_jwt = false`. While some (like `stripe-webhook`) need this, functions like `generate-story` and `create-checkout-session` should verify JWT to prevent unauthorized usage and cost abuse.

### 12. Missing `og:image` Meta Tags
`index.html` has `og:title` and `og:description` but **no `og:image`**. Social shares will look bare without a preview image.

### 13. `canonical` Tag is Relative
`index.html` line 9: `<link rel="canonical" href="/" />` -- should be an absolute URL for proper SEO.

## LOW Priority / Cosmetic

### 14. "The StoryMaster is weaving your tale..." Loading Text
Mission.tsx line 993 still says "The StoryMaster" instead of the branded name.

### 15. Disabled Abilities Code Cluttering Codebase
Hundreds of lines of commented-out abilities code throughout Mission.tsx, Dashboard.tsx, etc. While not a bug, it increases maintenance burden and file size.
