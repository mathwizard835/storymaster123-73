# Task: Harden Story Generation Reliability & Eliminate "Try That Choice Again" Loops

## Objective

Fix the issue where tapping a choice intermittently fails due to malformed AI output (missing choice buttons) or stream interruptions. We want a resilient, idempotent flow with server repair, silent client retry, and smooth UI transitions suitable for StoryMaster Kids.

---

## 1. Edge Function Update `supabase/functions/generate-story/index.ts`)

### A. Add Strict Scene Schema Validation

Define a helper function to validate scene completeness:

- Verify `sceneTitle`, `hud`, and `narrative` exist and are valid strings.

- If `isEnding` is `false` or undefined, `choices` MUST be an array with **exactly 3 items**.

### B. Add Turn Idempotency

- Check incoming request payload or headers for a unique `turnId` string generated by the client.

- Before calling the AI model, check if a scene entry for this `turnId` already exists in the database.

- If it exists, return the cached scene immediately to prevent double DB writes, double XP awards, or resource deductions.

### C. Server-Side Single Fast Repair Retry

- If the initial LLM completion fails schema validation or JSON parsing:

  - Trigger **at most ONE** fast repair call to the LLM.

  - Pass a repair prompt explicitly requesting JSON matching the required schema with 3 choices.

  - Apply a reduced `max_tokens` target to ensure the repair call finishes within 4–6 seconds.

- If the repair call still fails, return a clean HTTP `422 Unprocessable Entity` error.

---

## 2. API Helper & State Management `src/lib/story.ts`)

### A. Pass Unique `turnId` and `AbortController`

- Update the generation function to accept a client-generated `turnId` (e.g., via `crypto.randomUUID()`) and an `AbortSignal`.

- Ensure fetch calls pass the `turnId` in the request payload and bind the `AbortSignal`.

### B. Synchronous Cache / Promise Cleanup

- Ensure any `inFlightScenes` entries or pending promises are evicted synchronously in a `finally` block when a request fails or is aborted.

---

## 3. Mission Page & UI Retry Flow `src/pages/Mission.tsx`)

### A. Implement Single Silent Client Fallback

On choice tap `onChoose`):

1. Generate a new `turnId`.

2. Execute **Attempt 1** using streaming generation.

3. If Attempt 1 encounters a transient error (JSON parse failure, 422 schema error, or stream drop):

   - **Do NOT show the red "Try that choice again" toast yet.**

   - Immediately execute **Attempt 2** using a non-streamed fetch request, passing the **same `turnId**`.

4. Only display the red error toast if **both** Attempt 1 and Attempt 2 fail.

5. Immediately surface non-transient errors (e.g., auth, paywall, or rate limits) without retrying.

### B. Smooth UX (Prevent Screen Blanking)

- When Attempt 1 fails, **do not wipe `streamedNarrative**`.

- Keep partial narrative text on screen, dim its opacity slightly, and display a gentle, kid-friendly indicator like *"Summoning choices..."* or *"Writing the next chapter..."*.

- On Attempt 2 success: Smoothly swap out partial text with the verified scene and activate choices.

- On Attempt 2 failure: Revert UI state cleanly and display the error toast.

2. Verify duplicate choice taps using the same `turnId` do not grant duplicate XP or create duplicate database turns.

3. Simulate malformed output/network drops to confirm the app silently recovers without screen flickering.
